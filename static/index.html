01 What We Built
Crisis Room is an OpenEnv-compliant RL environment where an LLM agent learns to manage cascading production system failures — investigating root causes, issuing remediation commands, and communicating to stakeholders — all within a step budget that simulates real time pressure.

The environment exposes 3 difficulty levels, 8 action types, 4 independent reward components, and a full anti-reward-hacking suite. Runs as a FastAPI server, validates cleanly with openenv validate, and is deployed live on HF Spaces.
Incident Library — 6 Real Production Failures
EASY
DB Connection Pool Exhausted
Payment service returning 500 errors. Root cause visible in logs. Single service affected. Agent must check logs, identify pool exhaustion, restart the service.
Budget: 8 steps
EASY
SSL Certificate Expired
API gateway SSL handshake failures at 95%. Certificate expiry flagged in logs 7 days prior. Agent must run ssl_check diagnostic, then mark_resolved.
Budget: 8 steps
MEDIUM
Redis Out of Memory
Checkout, auth, and inventory all degraded simultaneously. Redis OOM taking down three services at once. Agent must identify the shared dependency, not treat each as separate.
Budget: 10 steps
MEDIUM
Memory Leak from Bad Deployment
Order service slowly degrading after a deployment. Memory growing at 120MB/min. No obvious crash — agent must correlate deployment timing with memory trend and rollback.
Budget: 10 steps
HARD
Silent Data Corruption
All services report healthy. No infrastructure alerts. Customers report wrong order amounts. Agent must run data_integrity diagnostic to find a float32 precision bug in v3.1.0.
Budget: 12 steps
HARD
Cascading DNS Failure
Intermittent failures across multiple services. CoreDNS configmap missing a forwarding entry after infra change. Agent must distinguish DNS failure from application errors.
Budget: 12 steps

03 Reward Function
Four independent components. No single signal can be gamed without the others pulling it down. 80% programmatic — reward-hack resistant.
Resolution
Did the agent fix the right service with the right action? Verified against hidden ground truth. services_restored ÷ services_affected.
40%
Investigation
Did the agent check relevant logs and run relevant diagnostics before acting? Measured against actual log keys in the incident. Rewards thorough investigation, not lucky guesses.
30%
Efficiency
Fewer steps = higher score. Resolving in 4 steps earns full marks. Burning all 12 steps earns near zero. Trains urgency without sacrificing accuracy.
20%
Communication
Was the team notified? Was the incident escalated on hard difficulty? Trains the communication habits that distinguish good incident commanders from great ones.
10%
−0.10per wrong restart, wrong rollback, or redundant notification
SPAM3 identical consecutive actions → automatic penalty. No free loops.
LOCKEDmax_steps set at reset from locked config — episodes cannot be extended
TRUNCATEDaction.target truncated to 500 chars — oversized payloads blocked
REWARD FORMULA
reward = (0.40 × resolution_score) # right service, right fix — vs hidden ground truth
       + (0.30 × investigation_score) # log keys + diagnostic names checked
       + (0.20 × efficiency_score) # fewer steps = higher score
       + (0.10 × communication_score) # notify_team + escalate on hard
       − (0.10 × wrong_restarts) # wrong fix / redundant action / unknown type
04 API Endpoints
POST/resetLoad a new incident scenario. Returns initial observation.
POST/stepSend an action. Returns observation, reward, done flag.
GET/tasksList available incident scenarios by difficulty.
GET/healthEnvironment health check — returns status + version.
# Start a hard incident
curl -X POST https://hemankkk-crisis_room.hf.space/reset \
  -H "Content-Type: application/json" \
  -d '{"difficulty": "hard"}'

# Run a diagnostic
curl -X POST https://hemankkk-crisis_room.hf.space/step \
  -d '{"action": {"action_type": "run_diagnostic", "target": "dns_check"}}'

# Resolve the incident
curl -X POST https://hemankkk-crisis_room.hf.space/step \
  -d '{"action": {"action_type": "restart_service", "target": "dns-resolver"}}'
